# 高考志愿推荐系统 · 演进方案

| 项 | 内容 |
| --- | --- |
| 文档版本 | v1.0 |
| 日期 | 2026-06-08 |
| 状态 | **已归档**——项目已重新定位为学习/作品集方向，现行规划见 [docs/项目规划.md](docs/项目规划.md)；本文档保留作为「转产品方向」时的路线参考 |
| 适用代码库 | `gaokao-volunteer-recommender-mvp`（React 19 + Express 5 + node:sqlite + Zod + TypeScript） |

> 一句话定位：当前系统是一个**结构正确的规则引擎骨架**，本方案的目标是把它演进为一个**准确、可解释、有安全护栏**的志愿决策助手。核心不是加更多按钮，而是补齐几个**领域纵深**。

---

## 1. 背景与现状

### 1.1 当前能力（基线）
- 输入考生分数/位次/选科/偏好，按"冲稳保兜底"分档输出候选院校专业。
- 规则引擎位于 [`src/recommendation.ts`](../src/recommendation.ts)，数据落 SQLite（[`server/db.ts`](../server/db.ts)），API 在 [`server/app.ts`](../server/app.ts)，前端工作台在 [`src/App.tsx`](../src/App.tsx)。
- 已有：来源留痕、免责声明、报告草案、管理导入、单元 + API 测试。

### 1.2 本轮已修复（评审基线）
- 推荐结果由"每档仅 1 条、全站最多 4 条"改为**返回完整分档列表**。
- 匹配纳入**选科（首选/再选科目）、生源省份、录取批次**三类硬约束。
- 风险分档由绝对位次差魔法数改为**位次相对差（比值）**，并接入风险偏好与服从调剂。
- 管理写接口加令牌守卫；输入边界校验；畸形请求体回 4xx；筛选逻辑前后端去重。

### 1.3 核心局限（为什么还是"玩具"）
| 局限 | 影响 |
| --- | --- |
| 分数↔位次用线性魔法系数估算 | 跨年、跨分段全面失真 |
| 无等效分/同位分换算 | 历年录取数据无法公平比较 |
| 数据模型是扁平"院校+专业" | 与新高考"院校专业组/专业+院校"填报单元不符 |
| 输出是孤立卡片、给模糊档位 | 用户真正要的是**整张有序志愿表 + 录取概率** |
| 无章程硬约束（单科/体检/语种） | 存在推荐了却会滑档/退档的风险 |
| 样例数据仅浙江视角 | 离真实多省可用尚远 |

## 2. 目标与设计原则

**目标**：从"规则引擎玩具"演进为"可用的志愿决策助手"，主线是 **准确性 → 概率化成表 → 安全护栏**。

**原则**
1. **准确性优先**：一切建立在扎实的位次法之上，宁可少推、不可误导。
2. **安全护栏**：高考是一锤子买卖，硬约束校验与退档风险是第一类公民。
3. **可解释**：每条推荐都附证据（历年位次、计划变化、概率推导、数据来源与时间）。
4. **隐私优先**：分数/位次为敏感信息，本地优先、非必要不上传。
5. **数据驱动 + 闭环校准**：用真实录取结果回测并持续校准模型。

## 3. 能力地图（7 大改进项）

> 每项标注：价值 / 领域背景 / 落到现有代码 / 粗估工作量（小团队人日，**数据获取为最大变量，单列**）。

### 3.1 整张志愿表 + 录取概率 ⭐ 最高杠杆
- **价值**：用户要的是有序志愿单（如浙江 80、院校专业组省份 45 组×6）+ 每条"录取概率 78%"，而非几张孤立卡片与模糊"稳"。
- **领域背景**：平行志愿"一次投档"，所以必须把志愿表当**有序组合**来优化——在"冲:稳:保:兜底 合理配比、城市/专业/院校适当分散、梯度不断档"约束下最大化整体质量。
- **落地**：`Recommendation` 增 `admissionProbability`；新增 `POST /api/plan` 生成成表草案 + 防滑档/断档体检；前端把卡片换成"志愿单 + 概率条"。
- **工作量**：概率 v1 约 2–3 人日；整表生成/优化 3–5 人日。

### 3.2 位次法地基：一分一段表 + 等效分换算 ⭐ 准确性根基
- **价值**：替换线性魔法系数，让所有下游（概率、成表）有可信输入。
- **领域背景**：分数↔位次必须**查表**（各省/各年/各科类一分一段表）；并做**等效分/同位分换算**——因每年难度、批次线、计划数都变，要把"今年位次"换算成"往年等效位次"再与历史录取线比。**这是位次法的灵魂**，缺了它跨年比较全错。还需**大小年识别**（录取线震荡时降低置信）。
- **落地**：新增 `score_segments` 表；新工具 `convertScoreToRank()` / `equivalentRank()`，替换 [`recommendation.ts`](../src/recommendation.ts) 的 `estimateRankFromScore`。
- **工作量**：引擎 1–2 人日；**数据获取为关键路径**。

### 3.3 志愿单元对齐新高考 + 多省配置化
- **价值**：模型与真实填报单元一致，才谈得上"多省支持"。
- **领域背景**：填报单元是**院校专业组**（物理/历史重组省份）或**专业(类)+院校**（浙江/山东），两者投档规则、志愿数、调剂范围完全不同。
- **落地**：`AdmissionRecord` → 志愿单元（含 `admissionGroup`、组内专业列表、调剂范围）；新增 `province_rules` 配置（模式/批次/志愿数/平行志愿规则），引擎按省切换。
- **工作量**：模型重构 3–5 人日。

### 3.4 硬约束校验 + 退档风险 ⭐ 安全护栏
- **价值**：最能体现"保护用户"的部分，目前几乎空白。
- **领域背景**：招生章程含**选科/单科分数/体检/外语语种/性别**等限制；**不服从调剂 + 院校专业分配规则（专业清/分数清/级差）+ 单科限制**会导致"进档却被退、滑到征集/下一批"。**滑档/退档是本场景最大痛点**。
- **落地**：新增 `restriction_rules` 库；`isRecordEligible` 升级为分级校验（硬拦截 vs 警示）；新增 `estimateRejectionRisk()`；前端对被拦截项给出明确原因。
- **工作量**：2–4 人日；**章程数据获取为变量**。

### 3.5 Claude 驱动的志愿规划助手（RAG）⭐ 差异化体验
- **价值**：干净的结构化数据正适合接**有据可查**的对话助手。
- **能力**：自然语言输入解析（"浙江物理 615 分 2 万名想学计算机、留长三角"→ 表单）；解释与追问（"为什么不推荐 XX""分不够计算机还能去哪些相关专业"），答案**逐条引用 source/updatedAt + 强制免责**，不编造。
- **落地**：`POST /api/assistant`，以 Claude 工具调用打到现有 `generateRecommendations` 与记录库；模型默认 `claude-opus-4-8`（或更省的 `claude-haiku-4-5`）。
- **工作量**：3–5 人日。

### 3.6 专业 / 职业维度
- **价值**：家长学生真正纠结的常是"学什么、将来干什么"，而非仅"能不能进"。
- **能力**：专业解读（培养方向、就业去向、薪资/考研率、转专业政策）；兴趣/职业测评（霍兰德等）→ 专业推荐，作为"不知道学啥"用户的前置漏斗。
- **落地**：新增 `majors_kb` 知识库表 + `/api/majors/:id`；测评模块独立。
- **工作量**：data-heavy，按数据源决定。

### 3.7 信任 / 合规 / 校准闭环
- **价值**：高考志愿是高风险 + 诈骗高发领域，可信本身即差异化。
- **能力**：全程可解释 + 每个数据点露出来源/时间/置信度；隐私（本地优先、不必要不上传）；**后验校准反馈环**——收集用户最终录取结果，回测"推荐档位 vs 真实命中率"，反向校准概率模型。
- **落地**：埋点 + `outcomes` 表 + 离线校准脚本；前端来源/置信度可视化。
- **工作量**：持续投入。

## 4. 优先级与路线图

三个阶段彼此咬合：**准确的位次换算是概率的输入，概率是成表的输入，硬约束是成表的安全网。**

| 阶段 | 主题 | 包含 | 产出 |
| --- | --- | --- | --- |
| **阶段一** | 地基 · 准确性 | 3.2 位次法/等效分；3.3 模型预重构 | 可信的分数↔位次换算、与新高考一致的数据模型 |
| **阶段二** | 核心体验 | 3.1 概率 + 整表生成；3.4 硬约束 + 退档 | 一键生成带概率、防滑档的完整志愿表 |
| **阶段三** | 差异化 | 3.5 Claude 助手；3.6 专业/职业；3.7 校准闭环 | 对话式规划、专业认知、持续自我校准 |

```
阶段一 ──> 阶段二 ──> 阶段三
位次法    概率+成表    Claude助手/专业/校准
(地基)    (主线体验)   (差异化, 可与二并行)
```

> **建议起步**：先做 **3.2 一分一段表 + 等效分换算**（替换线性魔法系数，给后面所有事打地基）；若想最快看到效果，可先上 **3.1 的录取概率字段**。Claude 助手（3.5）体验最出彩，可与阶段二并行。

## 5. 数据模型与接口演进（落地草图）

### 5.1 数据模型
- `AdmissionRecord` → **志愿单元**：新增 `admissionGroup`（院校专业组）、组内专业列表、`adjustmentScope`（调剂范围）、`assignRule`（专业清/分数清/级差）。
- 新表：
  - `score_segments(province, year, track, score, rank, cumulative)` —— 一分一段表。
  - `province_rules(province, mode, batches, slotCount, parallelRule)` —— 各省填报规则配置。
  - `restriction_rules(recordId, type, expr, level)` —— 章程硬/软限制。
  - `majors_kb(id, name, intro, careers, salary, postgradRate, transferPolicy)` —— 专业知识库。
  - `outcomes(anonId, planSnapshot, admittedRecordId, createdAt)` —— 后验校准。

### 5.2 接口
| 方法 | 路径 | 说明 |
| --- | --- | --- |
| POST | `/api/plan` | 生成完整有序志愿表草案 + 防滑档/断档体检 |
| POST | `/api/assistant` | Claude 规划助手（RAG，答案附来源 + 免责） |
| GET | `/api/majors/:id` | 专业解读 |
| POST | `/api/outcomes` | 上报最终录取结果（匿名，供校准） |
| —— | 现有 `/api/recommendations` 等 | 复用为 `/api/plan` 的底层能力 |

### 5.3 关键算法点
- **等效分换算**：今年位次 →（一分一段表）今年分 → 与历史按位次对齐 → 历史等效位次，用于和录取最低位次比较。
- **录取概率 v1（经验法）**：以"考生等效位次相对历年录取位次分布"的位置 + 计划变化 + 大小年因子，映射到 `P(录取)`；先用单调经验曲线，后续用 `outcomes` 校准。
- **整表优化**：约束满足 + 目标最大化——满足"冲稳保兜底配比、梯度连续不断档、维度分散"，最大化整体院校/专业质量。
- **退档风险**：`不服从调剂 × 专业分配规则 × 单科/体检限制` → 进档被退概率。

## 6. 风险与依赖

| 风险 | 说明 | 应对 |
| --- | --- | --- |
| **数据获取（关键路径）** | 一分一段表、历年录取、招生计划、章程限制均来自各省考试院/阳光高考，格式各异、更新有时效 | 优先搭 ETL 与版本化校验；先 1–2 省跑通再扩省 |
| 数据准确性与时效 | 错误数据会误导高风险决策 | 来源留痕 + 置信度 + 强免责；关键项人工复核 |
| 合规与隐私 | 敏感成绩信息 | 本地优先、最小化上传、明确隐私声明 |
| 模型可信度 | 概率被误读为承诺 | 显式区间 + 解释 + "不构成录取承诺" |
| 数据库迁移 | 新增列，`CREATE TABLE IF NOT EXISTS` 不会补列 | 引入正式迁移机制（当前靠删库重建仅适用于开发） |

## 7. 术语表

| 术语 | 含义 |
| --- | --- |
| 一分一段表 | 各省按分数公布的人数累计表，用于分数↔位次互查 |
| 位次法 | 以全省位次（而非裸分）跨年比较录取可能性的方法 |
| 等效分/同位分 | 把今年位次换算成往年等效分/位次，消除年度难度差异 |
| 院校专业组 | 新高考填报单元之一，一个院校按选科要求分成若干专业组 |
| 平行志愿 | "分数优先、遵循志愿、一次投档"的投档规则 |
| 大小年 | 院校录取线因考生博弈而高低交替的现象 |
| 退档/滑档 | 进档后因不服从调剂或限制被退、或未被任何志愿录取而落到下一批 |
