## 一、重新定位

**一句话定位**：一个展示「领域建模 + 全栈工程能力」的决策支持系统作品——不是 CRUD 演示，而是带真实领域算法（位次法、等效分换算、风险分档）和数据工程（ETL、迁移、版本留痕）的完整项目。

**目标读者**（按优先级）：
1. **面试官 / 招聘方**——平均停留 10 分钟：看 README、点 live demo、扫一眼代码结构
2. **技术同行**——可能读核心算法和测试
3. 自己——学习沉淀

**核心叙事**（作品集要讲的三个点）：
1. **领域算法**：位次法为什么比裸分数对比正确？等效分怎么换算？大小年怎么识别？——有深度、面试可展开
2. **数据工程**：CSV 导入管线（dry-run / merge / 版本留痕）、`PRAGMA user_version` 迁移、一分一段表入库注册——体现「为真实数据规模设计」的工程意识
3. **质量工程**：30+ 测试（单元 + API 集成）、类型全覆盖、CI、可复现的确定性样例数据

> 定位变化的直接推论：**演示样例数据从"缺陷"变成"特性"**——确定性生成、自洽、可复现，只需显著声明"非真实数据"。不再追求接入真实考试院数据（管线保留作为工程展示）。

## 四、技术架构目标状态

### 4.1 前端拆分（最大的代码债）
```
src/
├── components/        # App.tsx 拆出：InputPanel / FilterRail / ResultList /
│   ├── ...            #   RecommendationRow / DetailPanel / ComparisonPanel / ReportPanel
├── hooks/             # useRecommendations（API 状态机：加载/错误/数据）
├── engine/            # recommendation.ts + scoreRank.ts（领域层，独立可测）
├── api.ts / types.ts / data.ts
```
原则：领域层（engine）零 React 依赖——这是面试讲「分层」的素材。

### 4.2 部署形态
- **单容器**：Node 22+（node:sqlite 需要）跑 Express，同时托管 `vite build` 产物，SQLite 文件随容器卷
- 平台任选 Fly.io / Railway / Render（免费层够用）
- 生产配置：CORS 收紧到同源、基础限流、`GAOKAO_ADMIN_TOKEN` 必设

### 4.3 CI
GitHub Actions：push → typecheck + test + build；徽章放 README。
